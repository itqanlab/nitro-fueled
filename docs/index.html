Step 5 · Worker Roles
Autonomous Worker Sessions
The Supervisor spawns isolated worker sessions via MCP. Each worker runs a full orchestration pipeline in its own context.
⚙

Build Worker

Runs the full PM → Architect → Dev pipeline. Takes a task from CREATED through implementation. Commits code and marks IMPLEMENTED.

CREATED → IMPLEMENTED
🔎

Review Lead

Spawns three parallel sub-workers: Style Reviewer (Sonnet), Logic Reviewer (Opus), Security Reviewer (Sonnet). Collects results, runs fix phase, appends review lessons, writes completion report. Marks COMPLETE.

IMPLEMENTED → COMPLETE
🔨

Cleanup Worker

Lightweight salvage session. When a worker dies, this commits uncommitted work and updates registry status. Fast in, fast out.

Salvage → Commit → Exit
Step 6 · Control Loop
Auto-Pilot Core Loop
The Supervisor coordinates workers end-to-end: queue selection, spawning, monitoring, recovery, and stop conditions.
0

Pre-Flight Validation

Runs before the loop starts. Checks dependency satisfiability, circular deps, file overlap, task completeness, MCP health, and task sizing. Blocking issues abort; warnings log and proceed.

1

Read State & Recover

Read session state directory, reconcile active workers against MCP, validate registry states. Recover from compactions.

2

Build Dependency Graph

Parse registry and task.md files. Classify tasks as READY_FOR_BUILD, READY_FOR_REVIEW, or BLOCKED. Detect dependency cycles.

3

Consult Strategic Plan

Read plan.md for Supervisor Guidance. Apply PROCEED, REPRIORITIZE, ESCALATE, or NO_ACTION. Use plan ordering to break ties.

4

Order & Spawn

Review queue first (finish > start). Then build queue. Fill available slots up to concurrency limit. Spawn via MCP.

5

Monitor Health

Every 5 minutes: check worker activity, escalate to full stats if issues detected. Two-strike stuck detection.

6

Handle Completions

Validate state transitions. Build done? Queue review. Review done? Mark complete. No transition? Spawn Cleanup Worker, then retry.

7

Recovery Protocol

Worker died? Spawn Cleanup Worker to salvage uncommitted work. Re-read registry. Spawn replacement worker or mark BLOCKED.

8

Loop or Stop

Actionable tasks? → Loop. Active workers only? → Wait. Nothing left? → Save history, stop. Max 2 compactions.

Step 7 · Decision Engine
How Orchestration Chooses Skills vs Agents
Commands first route intent, then the orchestrator decides whether to execute a skill workflow, invoke a specialist agent, or chain both.

User Input

Command (`/plan`, `/orchestrate`, `/review-*`) + task context

Command Router

Loads relevant task files, state, and requested mode

Decision Gate

Classifies request as workflow-oriented, specialist-oriented, or hybrid

↓
Skill Path

Use when work requires a repeatable multi-step process with checkpoints (for example orchestration loop, content workflow, or design workflow).

Agent Path

Use when work needs deep role expertise (for example Architect, Team Leader, reviewer, or framework-specific developer).

↓

Execution Outcome

Skills can call agents, agents can run skills, and Supervisor workers enforce state transitions and deliverables.

Step 11 · Standards
Tracking & Control Layer
Every task produces structured artifacts. These files are the system's memory — they give the Supervisor, reviewers, and you full observability into what happened, why, and what's next.

Task Folder Structure

Each task lives in its own folder under task-tracking/. The Supervisor and workers read and write these files at every stage.

📁 task-tracking/
■ registry.md Central task index — status, type, dates
■ plan.md Phased roadmap, milestones, decisions log
■ task-template.md Canonical template for new tasks
■ sizing-rules.md Shared task sizing limits — enforced at creation and pre-flight
📁 sessions/
📁 SESSION_yyyy-mm-dd_HH-MM/
■ state.md Live Supervisor state — workers, queue (per-session)
■ log.md Unified event log — all orchestration paths write here
■ analytics.md Post-session analytics
📁 worker-logs/ Worker logs scoped to this session
📁 TASK_2026_NNN/
■ task.md Task definition — type, priority, acceptance criteria
■ context.md PM-generated requirements and scope
■ implementation-plan.md Architect's design decisions and file targets
■ tasks.md Team Leader's developer batch breakdown
■ code-style-review.md Style reviewer findings
■ code-logic-review.md Logic reviewer findings
■ security-review.md Security reviewer findings
■ completion-report.md Final summary — what shipped, tradeoffs, follow-ups

Control & Observability Files

These top-level files form the control plane. The Supervisor reads them every loop iteration to make decisions.

registry.md
Single source of truth

Central index of every task: ID, status, type, description, and creation date. The Supervisor's first read on every loop. All state transitions are recorded here.

Read by: Supervisor, Planner, /project-status
plan.md
Strategic roadmap

Phased project plan with milestones, task maps, and a decisions log. Contains Supervisor Guidance directives (PROCEED, REPRIORITIZE, ESCALATE) that steer the auto-pilot loop.

Written by: Planner · Read by: Supervisor
sessions/SESSION_xxx/
Per-session state directory

Replaces the single orchestrator-state.md. Each run gets its own directory with state.md (live worker queue), log.md (unified event log all orchestration paths write to), and analytics.md. Enables concurrent supervisors and full history retention.

Written by: Supervisor · Read by: Supervisor, /project-status
task-template.md
Task creation standard

Canonical template with metadata fields (type, priority, complexity), description, dependencies, acceptance criteria, and references. Ensures every task is machine-parseable by the Supervisor.

Used by: /create-task, Planner, PM
anti-patterns.md
QA-derived rules

Rules extracted from 142+ QA findings across completed tasks. Covers file size limits, resource cleanup, silent failures, race conditions, and framework-specific pitfalls. Checked before every code submission.

Read by: All developers, all reviewers
review-lessons/
Per-domain learned patterns

Growing library of reviewer-discovered patterns split by domain: backend, frontend, and general. Each review worker appends new lessons, so knowledge compounds across tasks.

Written by: Reviewers · Read by: Developers, Reviewers

Staying Current
How It Stays Current
Nitro-Fueled uses a manifest-based update model to keep your agents sharp without overwriting project customizations.
nitro-manifest.json
Version registry

Generated at init time. Records the version of every installed agent, skill, and command file. npx nitro-fueled update diffs against the latest manifest to find what has changed upstream.

Written by: CLI init/update · Read by: CLI update
Core agents (nitro-*)
Always replaceable

All nitro-* agents ship verbatim from the package. The nitro- prefix reserves this namespace — any file matching .claude/agents/nitro-*.md is safe to overwrite on update.

Updated by: npx nitro-fueled update
Project agents
Your customizations, preserved

Agents generated at init time for your stack (e.g., frontend-developer.md, backend-developer.md) do not carry the nitro- prefix. They are never overwritten by update.

Generated by: npx nitro-fueled init · Owned by: your project
review-lessons/ & anti-patterns.md
Project-grown knowledge

These files start empty (or minimal) and accumulate QA findings specific to your codebase. They are never touched by update — only your review workers write to them.

Written by: Review Workers · Read by: All developers, reviewers
