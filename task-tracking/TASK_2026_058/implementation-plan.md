# Implementation Plan — TASK_2026_058

## Architecture Decision

The core insight is that `registry.md` currently serves two purposes that are now being separated:

1. **State source of truth** — what phase is each task in? (moving to per-task `status` files)
2. **Human-readable summary** — a table of all tasks with metadata (staying in `registry.md`, but as generated output)

The new design:
- `task-tracking/TASK_YYYY_NNN/status` — one word, no trailing newline, the live state for that task
- `registry.md` — a derived artifact regenerated by scanning all `TASK_*/status` + `TASK_*/task.md` files, with a `<!-- DO NOT EDIT — generated by nitro-fueled status -->` header

Workers write only to their own task folder. Zero cross-task conflict surface. The CLI `status` command and `/project-status` command become the regeneration entry points.

## Implementation Phases

### Phase 1: Backfill Existing Task Folders

This must happen first so the system is never in a state where `status` files are partially present. Once this phase is committed, the `status` file is the ground truth for all folders.

**Step 1.1: One-time migration script (manual execution)**

For each of the 60 existing `TASK_*/` folders in `task-tracking/`, write a `status` file containing the current status from `registry.md`. The exact status value comes from column 2 of each registry row.

This is done by the developer manually reading `registry.md` and writing the corresponding `status` file into each task folder. No trailing newline. Plain text, one word.

Example output: `task-tracking/TASK_2026_001/status` contains `COMPLETE`.

Valid values: `CREATED | IN_PROGRESS | IMPLEMENTED | IN_REVIEW | FIXING | COMPLETE | FAILED | BLOCKED | CANCELLED`

**Step 1.2: Verify backfill completeness**

After writing, confirm every task folder has a `status` file:

```bash
# Count task folders
ls -d task-tracking/TASK_*/ | wc -l

# Count status files
ls task-tracking/TASK_*/status | wc -l

# Both should match
```

**Step 1.3: Add `status` format to generated registry header**

Update `registry.md` to prepend:

```
<!-- DO NOT EDIT — generated by nitro-fueled status -->
```

This signals that the file is now a derived artifact.

### Phase 2: CLI `status` Command — Registry Regeneration

The CLI `packages/cli/src/commands/status.ts` becomes the authoritative regenerator of `registry.md`. When `nitro-fueled status` runs, it scans all `TASK_*/status` + `TASK_*/task.md` files and rewrites `registry.md` before displaying the report.

**Step 2.1: Add `generateRegistry()` to `packages/cli/src/utils/registry.ts`**

New function that:
1. Globs `task-tracking/TASK_*/` directories (sorted by Task ID)
2. For each folder, reads `status` file (falls back to reading `registry.md` row if `status` missing — graceful degradation during migration)
3. Reads `task.md` for metadata: Type, Description, Created date, Model
4. Assembles a registry row
5. Writes `task-tracking/registry.md` with the generated header and full table

Function signature pattern (following the existing utility style in `registry.ts`):

```typescript
export function generateRegistry(cwd: string): void
```

**Step 2.2: Modify `registerStatusCommand` in `packages/cli/src/commands/status.ts`**

Before calling `parseRegistry(cwd)`, call `generateRegistry(cwd)`. This ensures the registry is always fresh when the status command runs.

Change order:
1. `generateRegistry(cwd)` — scan status files, rewrite registry.md
2. `parseRegistry(cwd)` — read the freshly generated registry.md
3. Render output (unchanged)

**Step 2.3: Update `parseRegistry()` to handle the generated header**

The generated header `<!-- DO NOT EDIT ... -->` is a comment line. Confirm the existing parser skips it (it already skips non-matching lines, so the comment will be ignored). No code change needed unless testing reveals otherwise.

## Migration Strategy

### Ordering Constraint

Phase 1 (backfill) MUST complete and be committed before any other phase. The system must never be in a state where some code reads `status` files that don't exist yet.

### Safe Transition Window

Because `registry.md` still exists and still has accurate data at the start of this task:
- The backfill reads from `registry.md` (which is still live/accurate)
- After backfill is committed, `status` files are ground truth
- Code changes in Phases 2-8 can then safely switch readers to `status` files

### Fallback During Implementation

The CLI `generateRegistry()` and auto-pilot Step 2 should include a graceful fallback: if a `status` file is missing for a task, read from `registry.md` column 2 and log a warning. This handles any task folder that was created after backfill but before the create-task command was updated.

### Post-Implementation State

After all phases complete:
- `registry.md` is regenerated each time `nitro-fueled status` or `/project-status` runs
- Workers write only to `task-tracking/TASK_YYYY_NNN/status`
- `registry.md` retains the generated header, signaling it is not to be hand-edited

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Backfill misses a task folder | Low | Verify count: `ls TASK_*/status \| wc -l` must equal `ls -d TASK_*/ \| wc -l` |
| Worker writes `status` file with wrong value or trailing newline | Low | File content validation in `generateRegistry()` — trim whitespace before using value |
| Concurrent workers writing to same `status` file | Zero | Each task has exactly one Build Worker at a time (Supervisor enforces this) |
| `registry.md` read for ID generation gives wrong highest ID | Low | Registry still has all task IDs in column 1; ID generation reads column 1 only, unaffected |
| Scaffold files diverge from source | Medium | Phase 8 is last phase; confirm scaffold files are full copies before editing |
| FIXING state missing from `status` valid values | Medium | The FIXING state (used by Fix Worker) must be in the valid values list in `registry.ts`'s `VALID_STATUSES` and in the `generateRegistry()` function. Confirm it is added. |
| `parseRegistry()` regex doesn't match new header line | None | Regex requires `| TASK_...` prefix; comment line skipped naturally |
| Plan.md staleness detection breaks | Low | Planner §5d reads `status` files directly; plan.md comparison logic unchanged |
