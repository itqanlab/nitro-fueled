## Description

Replace `registry.md` as the live source of truth for task state with per-task `status` files. Currently, workers read and rewrite `registry.md` to update task status. This causes two problems: (1) concurrent or sequential workers with stale reads silently overwrite each other's status updates, and (2) as the registry grows, reading and rewriting the full file costs increasing context on every single status change.

The new design: each task folder contains a single `status` file (one word). Workers write only to their own task folder — zero cross-task conflicts, minimal context. `registry.md` becomes a generated artifact with a "DO NOT EDIT — generated" header. `nitro-fueled status` and `/project-status` regenerate it by scanning all `TASK_*/status` + `TASK_*/task.md` files. There is one source of truth; the registry is a derived view.

### Implementation Details

**1. Per-task status file**
- Path: `task-tracking/TASK_YYYY_NNN/status`
- Content: one of `CREATED | IN_PROGRESS | IMPLEMENTED | IN_REVIEW | FIXING | COMPLETE | FAILED | BLOCKED | CANCELLED`
- Plain text, no frontmatter, no trailing newline

**2. Backfill existing task folders**
- One-time migration: for every existing `TASK_*/` folder, write the current status from `registry.md` into a `status` file

**3. Orchestration skill — completion bookkeeping**
- Remove the `registry.md` write step entirely
- Replace with: write the new state to `task-tracking/TASK_YYYY_NNN/status`

**4. Auto-pilot skill — state machine reads**
- Replace all `registry.md` state reads with reads of `TASK_*/status` files
- Continue reading `task.md` for metadata (type, priority, dependencies) — never for state

**5. Planner agent**
- Replace registry.md state reads with `TASK_*/status` file reads
- Use registry.md only as a human-readable summary; never rely on it for decisions

**6. CLI `status` command + `/project-status` command**
- Regenerate `registry.md` by scanning all `task-tracking/TASK_*/status` and `task-tracking/TASK_*/task.md`
- Add `<!-- DO NOT EDIT — generated by nitro-fueled status -->` header to registry.md
- Scaffold copy in `packages/cli/scaffold/` must match

**7. `/create-task` command**
- After creating the task folder, write `status` file with `CREATED`
- Remove the registry.md row-append step — registry is regenerated, not appended to
