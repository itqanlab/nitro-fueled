## Score: 5/10

## The 5 Critical Questions

### 1. What could break in 6 months?

The `--regen` manifest update gap (see Critical #1) will cause silent state divergence. After `handleRegen` regenerates `anti-patterns.md`, the manifest is never updated with the new checksum or timestamp because `writeManifest` is called at Step 6 and `handleRegen` runs at Step 7. The next `update` run will compute a checksum mismatch against an obsolete manifest entry and treat the freshly-regenerated file as "user-modified" ‚Äî skipping it permanently. `update.ts:284-299`.

### 2. What would confuse a new team member?

The `--yes` flag is declared in `UpdateOptions` and registered as a CLI option, but `opts.yes` is never read anywhere in the action handler. A developer adding interactive prompts to `update` will look for how to handle `--yes` and find it in the interface and the option registration ‚Äî but it does nothing. There is no comment explaining this. `update.ts:34, 226`.

The `--show-diff` flag is mentioned in output to the user (`‚Üí run with --show-diff to review changes`, line 166) but is never registered as an option on the command. Running with `--show-diff` will have no effect or produce a Commander.js unknown-option error. This is a broken user-facing instruction.

### 3. What's the hidden complexity cost?

`getPackageVersion()` and `hasVersion()` are copy-pasted verbatim from `init.ts`. Both files instantiate their own `createRequire` (named `updateRequire` and `initRequire` respectively) just to read the same `package.json`. This means the next change to version-reading logic must be made in two places. There is no shared utility for this pattern. `update.ts:11-30`, `init.ts:20-37`.

### 4. What pattern inconsistencies exist?

`walkScaffoldFiles` uses a nested named function `walk()` closure to accumulate into a `Map` from the outer scope. The existing `copyDirRecursive` in the same file is a standalone recursive function that accepts `result` as a parameter and returns it. These two approaches in the same file are inconsistent ‚Äî the implicit state capture in `walk()` is harder to follow than the pure return value style used everywhere else in `scaffold.ts`.

`update.ts` variable naming: `opts.dryRun` follows Commander's camelCase transform from `--dry-run`, which is correct. But the `UpdateOptions` interface uses `dryRun` while `initOptions` uses `overwrite` and `skipMcp` ‚Äî both are consistent with camelCase, so no violation. However the `opts.yes` field is live in the interface but dead in use, creating a false consistency signal.

### 5. What would I do differently?

- Extract `getPackageVersion()` and `hasVersion()` into a shared `packages/cli/src/utils/version.ts` module. Both `init.ts` and `update.ts` import it from one place.
- Reorder operations so `handleRegen` (Step 7) runs before `writeManifest` (Step 6), or update the manifest after regen completes.
- Either implement `--show-diff` or remove the console hint. If it's future work, annotate it as a `// TODO: --show-diff not yet implemented`.
- Either remove `--yes` or add a comment explaining it is reserved for future prompts.

## Findings

### Critical (must fix)

**C1: `--regen` runs after manifest write ‚Äî regenerated files never tracked**

`update.ts:284-299`

The manifest is written at Step 6 (`writeManifest`, line 287). `handleRegen` runs at Step 7 (line 298), after the write. Any file regenerated by `handleRegen` (e.g., `anti-patterns.md`) will not have its checksum updated in the persisted manifest. On the next `update` invocation, the freshly-regenerated file's checksum will differ from the stale manifest entry, and `processScaffoldFiles` will classify it as "user-modified" and skip it permanently. This is an invisible data integrity bug ‚Äî no error is emitted, and the user has no way to recover without manually deleting the manifest.

Fix: Move `handleRegen` before `writeManifest`, or call `updateManifestData` a second time after regen and write the manifest again.

**C2: `--show-diff` referenced in output but never registered as a flag**

`update.ts:166`

```
console.log(`      ‚Üí run with --show-diff to review changes`);
```

`--show-diff` is not registered with `.option(...)` on the command. Running `npx nitro-fueled update --show-diff` will either be silently ignored or cause Commander.js to error with "unknown option". This is a broken user-facing instruction that ships in every `update` run that has skipped files.

Fix: Register the flag (even as a no-op with a `// TODO` comment), or remove the hint from the output until the feature is implemented.

### Major (should fix)

**M1: `getPackageVersion` and `hasVersion` duplicated from `init.ts`**

`update.ts:11-30`, `init.ts:20-37`

Both functions are identical (modulo the `createRequire` variable name). Any change to the version-reading logic must be applied twice. This violates the DRY principle and will cause a divergence bug when one copy is patched and the other is not.

Extract to `packages/cli/src/utils/version.ts` and import from both commands.

**M2: `opts.yes` declared, registered, and never used**

`update.ts:34, 226`

`UpdateOptions.yes: boolean` is declared and `--yes` is registered as a CLI option. But `opts.yes` is never referenced in the action handler. There are no interactive prompts in `update` today, so the flag is inert. A developer adding a confirmation prompt will discover it in the interface and assume it is wired up ‚Äî it isn't.

Fix: Either add a `// Reserved for future interactive prompts` comment inline with the option registration, or remove both the interface field and the option registration until it is needed.

**M3: `processScaffoldFiles` silently skips `generatedFiles` with no user feedback**

`update.ts:77-79`

When a scaffold file path matches a key in `manifest.generatedFiles`, it is silently `continue`d. The user running `update` sees nothing about these files ‚Äî they are not listed in the printed output, not in the skipped count, and not in the "Generated files (not touched)" block in `printResults`. The generated files block at line 179 only lists what is in `manifest.generatedFiles`, not which scaffold files were skipped because of it.

This means if a future scaffold release replaces a scaffold file with a key that collides with a generated file entry, the user cannot tell from `update` output that the replacement was suppressed.

Fix: Either include silently-skipped generated/scaffold-collision paths in `printResults`, or add a note to the "Generated files (not touched)" block explaining that scaffold updates for these paths are also suppressed.

**M4: `updateManifestData` mutates the `manifest` argument in-place without documentation**

`update.ts:114-140`

The function signature is `updateManifestData(manifest: Manifest, ...) : void`. It directly mutates `manifest.version`, `manifest.updatedAt`, `manifest.coreFiles`, and deletes keys from `manifest.coreFiles`. The name does not make the mutation apparent ‚Äî "update" in CLI context reads as "does the update operation". A future maintainer extracting this function must know to pass the live reference, not a copy.

Fix: Add a JSDoc comment: `/** Mutates manifest in-place: updates version, updatedAt, coreFiles. */`. Alternatively, return the updated manifest and assign at the call site to make the mutation visible.

**M5: `walkScaffoldFiles` uses closure accumulation ‚Äî inconsistent with sibling function style**

`packages/cli/src/utils/scaffold.ts:106-127`

`copyDirRecursive` returns a `CopyResult` and threads it through recursive calls explicitly. `walkScaffoldFiles` instead defines a nested `walk()` closure that mutates the outer `result` Map. These two recursive patterns in the same file create cognitive dissonance ‚Äî a contributor reading `walkScaffoldFiles` will expect the same return-value style established by `copyDirRecursive`.

The closure approach is not wrong, but the inconsistency means the next recursive function added to this file may go either way. Establish one pattern.

### Minor (nice to have)

**m1: Magic number `55` for column padding repeated five times**

`update.ts:156, 159, 163, 165, 184`

`r.relPath.padEnd(55)` appears five times in `printResults`. If the column width ever needs changing, five sites must be updated. Extract to a constant: `const COL_WIDTH = 55`.

**m2: `ensureParentDir` is a one-shot private helper with a generic name**

`update.ts:45-50`

This function exists only to be called from `copyFile`. Its name `ensureParentDir` is generic enough to suggest it belongs in a shared utilities module ‚Äî but it isn't exported and its only caller is the local `copyFile`. The indirection adds cognitive overhead without value. Inline it into `copyFile`, or if it is expected to grow, move it to a shared utils module.

**m3: `dryRun` guard embedded in `copyFile` but manifest write guarded externally**

`update.ts:52-56, 284`

`copyFile` handles `dryRun` internally (line 53: `if (dryRun) return`). But the manifest write at line 284 is guarded by `if (!opts.dryRun)` in the action handler. The two dry-run guard locations are asymmetric ‚Äî one is hidden inside a helper, the other is explicit at the call site. This is not a bug but will confuse the next person adding a dry-run-aware operation.

**m4: `updateRequire` / `initRequire` naming is a code smell from copy-paste**

`update.ts:11`

```typescript
const updateRequire = createRequire(import.meta.url);
```

The `updateRequire` name exists only to differentiate from `initRequire` in `init.ts`. If both lived in a shared module there would be no naming conflict. The rename is a side effect of the duplication and points to the root problem at M1.

**m5: `console.error` used for the `--show-diff` hint instead of `console.log`**

`update.ts:165-166`

The skipped-file hint `‚Üí run with --show-diff to review changes` is written with `console.log`. This is fine for user output. But the pattern is inconsistent in the broader file ‚Äî some informational messages use `console.log`, errors use `console.error`. The hint itself is not an error; the current `console.log` usage is correct. No action needed ‚Äî but confirm this is intentional (it is).
