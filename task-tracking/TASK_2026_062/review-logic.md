### Logic Issues

#### Issue 1 (Critical): `wrong_tool_used` tag is undetectable by scoring

**Step**: Step 6 (scoring dimensions)

**Problem**: The failure taxonomy defines exactly 4 tags: `scope_exceeded`, `instruction_ignored`, `quality_low`, `wrong_tool_used`. Step 4 correctly maps all 4 to test designs. Step 7a correctly maps all 4 to fix strategies. But Step 6 only has 3 scoring dimensions, and each dimension maps to exactly one tag:

- Dimension 1 (Scope) → `scope_exceeded`
- Dimension 2 (Instructions) → `instruction_ignored`
- Dimension 3 (Quality) → `quality_low`

There is no scoring path that produces `wrong_tool_used`. If an agent's `TOP_FAILURE_TAG` is `wrong_tool_used`, the test scenario is correctly designed to exercise unauthorized tool use (Step 4), but even if the agent uses a wrong tool during evaluation, Step 6 cannot tag it. The wrong_tool_used failure will either be silently missed or mis-tagged as `scope_exceeded` (which is adjacent but not the same). The fix path in Step 7a for `wrong_tool_used` is unreachable.

**Fix**: Add a 4th scoring dimension to Step 6: "Dimension 4: Tool Use Check — did the agent use tools or approaches outside its expected workflow?" with a pass/fail and tag `wrong_tool_used`.

---

#### Issue 2 (Critical): Retroactive edit of eval block violates append-only rule

**Step**: Steps 7 and 7a

**Problem**: Step 7 instructs writing the eval block to the record. Step 7a then applies the fix to the agent definition and says "update the `Changes made:` field in the eval block you just wrote." This requires going back and editing the block that was just appended — a retroactive mutation.

The schema in `agent-calibration.md` states an explicit append-only rule: "Never delete or edit existing rows." The command's two-phase write (write block, then edit the block's field) directly contradicts this rule. An interrupted execution after Step 7 but before the retroactive update leaves the record with `Changes made: [description]` not yet populated, or worse, the fix is applied to the agent definition but the record still says "Changes made: none".

**Fix**: Either (a) apply the agent definition fix BEFORE writing the eval block so `Changes made:` can be populated in the initial write, or (b) write a second, separate correction block per the schema's correction row convention rather than editing the existing block.

---

#### Issue 3 (Serious): Agent definition is patched on the 3rd iteration even though no re-run will use the patch

**Step**: Steps 7a and 7b ordering

**Problem**: The current flow on a 3rd consecutive failure is:
1. Write eval block (Iteration: 3 of 3, Result: FAIL)
2. Step 7a: apply fix to agent `.md` definition
3. Update `Changes made:` in eval block
4. Step 7b: increment ITERATION to 4, `4 > 3`, jump to Step 8 (FLAGGED)

The fix applied in 7a serves no purpose because no re-evaluation will follow — the agent is about to be flagged and the command stops. The agent definition gets a patch that was never validated by a subsequent run, and the FLAGGED summary in Step 8 describes what "was tried" across iterations, which now includes a 4th patch that never ran. This creates a misleading state: the definition has been modified but the modification is unproven.

**Fix**: Move the iteration limit check (currently 7b) to BEFORE Step 7a. If `ITERATION + 1 > 3`, skip the fix and jump to Step 8. Only apply the definition patch when a re-run will actually validate it.

---

#### Issue 4 (Serious): `PRIOR_CONSECUTIVE_FAILURES = 3` with `Status = ACTIVE` is a reachable inconsistent state

**Step**: Step 3 (FLAGGED guard) and Step 5 (pre-run guard)

**Problem**: If a previous evaluation run wrote 3 consecutive FAIL entries to the Evaluation History but crashed before executing Step 8 (which sets Status to FLAGGED), the agent record will have:
- `## Status`: ACTIVE
- 3 consecutive FAIL entries in Evaluation History

On the next invocation, Step 3's FLAGGED guard will pass (status is ACTIVE), and Step 3 item 3 will count `PRIOR_CONSECUTIVE_FAILURES = 3`. Step 5 sets `ITERATION = 4`, and the pre-run guard fires correctly, routing to Step 8. Step 8 then correctly flags the agent.

This is actually handled correctly by the Step 5 guard — it recovers from the inconsistency. However, Step 8 prints "What was tried: Iteration 1/2/3 fixes" but the command has no access to what those fixes were in prior sessions (they're in the eval history, but the summary template requires reconstructing them). The Step 8 summary instructs printing fix details across all 3 iterations, but for a recovered-from-crash scenario, only the current session's context is available.

This is a moderate gap in the summary completeness, not a data loss issue.

---

#### Issue 5 (Moderate): Step 7b message wording is ambiguous and error-prone

**Step**: Step 7b

**Problem**: After failing iteration N and incrementing ITERATION to N+1, the progress message is:

```
"Iteration {ITERATION - 1} of 3 FAILED. Re-running with updated definition..."
```

The executor must subtract 1 from the current value of ITERATION. This is a mental arithmetic step embedded in a message template. There is a risk of the wrong iteration number being printed (e.g., printing "Iteration 2 of 3 FAILED" when iteration 2 actually just ran and ITERATION is now 3). This is not a correctness bug in the loop itself, but in the user-facing feedback during the loop. The cleaner approach is to save the completed iteration number in a separate variable (e.g., `COMPLETED_ITERATION = ITERATION` before incrementing).

---

#### Issue 6 (Moderate): Eval block "Test task" format diverges from schema example

**Step**: Step 7

**Problem**: The command writes `Test task: inline — {one-line description}`. The schema in `agent-calibration.md` specifies `TASK_YYYY_NNN (or inline description if no task ID)`. The schema parenthetical is documentation guidance, not a literal prefix. The command introduces the literal token `inline — ` as part of the field value, which differs from the format shown in the schema's complete filled example (which uses a task ID directly). If parsing tools ever read this field to distinguish inline evals from tracked task evals, the prefix mismatch will cause issues.

---

#### Issue 7 (Minor): No handling for agents with a mix of old and recent PASS entries

**Step**: Step 3 item 3

**Problem**: "Count the number of consecutive `Result: FAIL` blocks before any `Result: PASS` block." If the eval history contains PASS entries followed by multiple FAIL entries (i.e., the agent passed previously but is now failing again), `PRIOR_CONSECUTIVE_FAILURES` correctly counts only the trailing fails. This is correct behavior. However, the command does not distinguish between "agent never passed" and "agent passed in a prior cycle then started failing again." The FLAGGED threshold of 3 applies equally in both cases, which is the intended behavior. No bug — just worth noting the implicit assumption.

### Summary

The command's overall structure is sound: the pre-flight checks are thorough, the FLAGGED guard is correct, the loop mechanics are mostly correct, and the failure taxonomy integration is coherent. The happy path (first-run, generic test, PASS) works cleanly.

However, there are two critical defects and one serious sequencing error that need to be addressed before this command is reliable in production:

**Critical — `wrong_tool_used` is a dead tag**: The 4th failure tag is part of the taxonomy, has a test design, has a fix strategy, but has no scoring dimension that can detect it. If an agent is known to use wrong tools, this calibration loop cannot catch or remedy that pattern. The 4th scoring dimension is simply missing from Step 6.

**Critical — Retroactive eval block edit violates append-only**: Writing the eval block before knowing what fix was applied forces a retroactive edit that contradicts the schema's stated append-only invariant. A crashed run between the two writes leaves the record in an inconsistent state. The fix ordering should be reversed: apply the definition patch first, then write the complete eval block in one atomic write.

**Serious — Agent definition patched on iteration 3 without a validating re-run**: The 3rd FAIL triggers both a definition patch and a FLAGGED status. The patch has no chance to be validated, creates a misleading entry in the summary, and leaves the definition in an unproven modified state. The iteration limit check should be moved before the fix application step.

The format compliance is mostly correct against the schema spec, with one minor divergence in the inline test task prefix notation.

Score reflects: core loop architecture is correct (credit), but 2 critical defects (wrong_tool_used blind spot, append-only violation) and 1 serious ordering bug reduce confidence materially. A production run targeting an agent with `wrong_tool_used` history will silently fail to calibrate that failure mode.
