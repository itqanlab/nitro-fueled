# Code Style Review — TASK_2026_044

## Score: 6/10

## Findings

### BLOCKING

- **Fallback table in Validation D omits the "Complexity + multiple layers" row from sizing-rules.md** — `auto-pilot.md:121` — The inline fallback limits table has four rows matching sizing-rules.md's four rows, but the fourth row ("Complexity + multiple layers") uses a different measurement description: the canonical file says "Split required" with no counting rule, while the fallback spells it out as `Complexity field is "Complex" AND description contains ≥2 of: service, layer, architecture, integration, pipeline`. That is actually good detail — but it diverges from the canonical source without noting the divergence. More critically, the review lesson at `review-general.md:56` is explicit: "A fallback that omits a dimension produces silent false-negatives in the one scenario (missing canonical file) where it is most needed." The fallback here does not omit a row, but it silently diverges on the detection method. A future edit to sizing-rules.md that changes the keyword list will not propagate to the fallback. The fallback must either match the canonical exactly or state explicitly that this inline expansion supersedes the canonical wording (which it cannot, since the canonical is the source of truth). As written, two implementations of "is this task complex enough to require a split" can produce different results depending on whether sizing-rules.md was available.

- **Validation B and C log format strings do not match the SKILL.md Session Log table** — `auto-pilot.md:97–99`, `SKILL.md:107–109` — The pre-flight abort path writes `[HH:MM:SS] PRE-FLIGHT FAILED — {N} blocking issue(s) found` (auto-pilot.md:157), which matches SKILL.md:92. But the Validation B blocking messages themselves (`"TASK_X: dependency TASK_Y not found in registry"`, `"TASK_X: dependency TASK_Y is FAILED — cannot proceed"`, `"TASK_X: dependency TASK_Y is CANCELLED — cannot proceed"`) are described only as items in `blocking_issues`, not as standalone session log entries. When these are eventually written to the session log (Step 4h), the format is not specified. The SKILL.md Session Log table defines three events that precisely cover these cases: `BLOCKED — TASK_X: dependency TASK_Y not in registry`, `BLOCKED — TASK_X: dependency TASK_Y is CANCELLED`, and `BLOCKED — TASK_X: dependency cycle with TASK_Y`. The pre-flight output messages use different wording (`"not found in registry"` vs `"not in registry"`, `"is CANCELLED — cannot proceed"` vs `"is CANCELLED"`). The review lesson `review-general.md:60` is unambiguous: "Enum values must match canonical source character-for-character." The session log is the canonical definition; the pre-flight report text should mirror it exactly or the divergence should be intentional and documented.

### SERIOUS

- **Step 4 opens with a behavioral note that belongs in a header comment, not inline prose** — `auto-pilot.md:69` — "Runs entirely in the command entry point — the Supervisor only receives a 'proceed' or 'abort' signal." This is critical architectural context, but it is buried as the first sentence of Step 4's body paragraph. A new contributor who reads only the step heading ("Pre-Flight Task Validation") will not understand the isolation contract. The review lesson `review-general.md:52` requires cross-file notes to be explicit; this note is the inverse — it tells a reader where this step does NOT run. It should be a prominent callout (blockquote or bold paragraph) immediately under the heading, not mixed into prose with the procedural steps.

- **Validation A "Description" counting rule is ambiguous for multi-line sentences** — `auto-pilot.md:88` — The rule "count sentence-ending punctuation followed by whitespace/newline" does not define what constitutes sentence-ending punctuation. A description field containing a list of bullet points ending with periods would pass (bullets end in `.`), while a two-sentence description with an ellipsis mid-sentence might fail or pass depending on the implementation. The review lesson `review-general.md:58` is specific: "Algorithm specs that count discrete items must define the counting rule precisely." The current rule leaves enough ambiguity that two implementations could disagree on whether `task.md` descriptions with embedded markdown lists satisfy the 2-sentence requirement. The rule should specify: count `.`, `!`, or `?` that are followed by a whitespace or newline AND are NOT inside a code fence or list item.

- **Validation C (circular dependency) does not log which specific tasks form the cycle in the session log** — `auto-pilot.md:108` — The spec says add to `blocking_issues`: `"Circular dependency detected: TASK_A -> TASK_B -> ... -> TASK_A"`. This appears in the PRE-FLIGHT VALIDATION REPORT output to the user. But Step 4h only says "Write... session log entry: `[HH:MM:SS] PRE-FLIGHT FAILED — {N} blocking issue(s) found`". The full cycle path is never written to the session log — only the count. If the user re-runs `/auto-pilot` in a new session without reading the terminal output, they cannot reconstruct which tasks formed the cycle from `orchestrator-state.md` alone. The session log should persist enough detail to diagnose issues post-compaction.

- **"Strict mode" referenced in task description is not implemented in the command** — `task.md:29`, `auto-pilot.md` (absent) — The task description says "proceed unless user configured strict mode." No `--strict` flag appears in the Parameters table or Step 2 argument parsing. This may be intentional scope reduction, but the omission is not acknowledged anywhere in the implementation. If the implementation deliberately defers strict mode, a note in Step 4 should say so explicitly, or the task acceptance criteria should reflect the reduction. As-is, a reader of the task.md expects strict mode to exist.

### MINOR

- **The pre-flight note in SKILL.md:235 references "Step 4" by number, not by name** — If the command entry point is ever reorganized and Step 4 is renumbered, this cross-reference silently breaks. The review lesson `review-general.md:68` states: "Cross-file section references must use names, not numbers." The reference should read: "The `/auto-pilot` command entry point (Step 4: Pre-Flight Task Validation) already ran..." — which adds the section name. Better still, it should read: "The Pre-Flight Task Validation step in `/auto-pilot` command entry point already ran..." — fully insulated from renumbering.

- **`sizing-rules.md:3` consumer list sentence is awkward with four consumers chained by commas** — The sentence now reads: "Referenced by `/create-task` (Step 6b), the Planner agent (Section 4c: Task Sizing Enforcement), the Planner **Backlog Sizing Review** (Section 3e), and `/auto-pilot` (Step 4: Pre-Flight Task Validation — Validation D)." This is correct but the inline parenthetical for `/auto-pilot` is notably longer than the others. Minor readability issue — consider using a bullet list for the consumer list if it grows further, since a prose list of four entries with embedded parentheticals is hard to scan.

- **Pre-flight report template uses `{CREATED_N}` and `{IMPLEMENTED_N}` variable names** — `auto-pilot.md:140` — These placeholder names are consistent internally but differ from the SKILL.md Summary display (Step 5) which uses `Ready for build (CREATED): {N}` and `Ready for review (IMPLEMENTED): {N}`. The variable naming convention is not standardized. This is cosmetic, but future contributors generating both outputs from the same data may introduce inconsistency.

- **Step 4a.1 says "already read in 3a — reuse" but Step 3a only says to verify existence, not necessarily to keep the content in memory** — `auto-pilot.md:73` — "Verify `task-tracking/registry.md` exists" (Step 3a) does not specify that the file contents are read into memory. Step 4a.1 assumes the registry was read at Step 3a, which is a safe assumption but not explicitly stated. If a future edit adds an explicit "read and parse" step at Step 3a, this note becomes redundant. If a future edit makes Step 3a a lightweight existence check only, Step 4a.1 breaks silently. The note should say "Read `task-tracking/registry.md` (or reuse if already in memory from Step 3)" to make the dependency explicit.

## Summary

The implementation is structurally sound and introduces a well-organized pre-flight step with five validations, a clear blocking-vs-warning classification, and proper session log integration. The three new Session Log rows in SKILL.md align with the existing table format. The consumer list addition in sizing-rules.md is correct.

The two blocking issues both stem from the same class of problem: specification text that diverges from the canonical definition it should mirror. The fallback table in Validation D expands the "Complexity" row without acknowledging the divergence from sizing-rules.md. The blocking issue messages use wording that differs from the SKILL.md Session Log table, which creates two definitions of the same event. Both are tractable fixes.

The serious issues are: architectural isolation context buried in prose (visibility), an ambiguous sentence-counting rule (precision), cycle details not persisted to the session log (debuggability), and the undocumented omission of strict mode (scope traceability).

The pre-existing `--interval` default mismatch (command says `10m`, SKILL.md says `5 minutes`) was not introduced by this task and is not counted against the score.
