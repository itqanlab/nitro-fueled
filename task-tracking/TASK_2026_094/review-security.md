# Security Review — TASK_2026_094

## Verdict
REQUEST_CHANGES

## Score
6/10

## Summary

The implementation is structurally sound — `MAX_QUEUE_SIZE` provides basic overflow protection, `label` is bounded to 64 chars, and the informational worker-existence check avoids race conditions with fast workers. However, two SERIOUS issues need addressing before merge: there is no payload size bound on the `data` field (making memory exhaustion trivially achievable despite the event count cap), and `worker_id` is not bound to the calling worker's identity (any caller can emit events under any worker's ID, enabling fake phase-transition injection). Three MINOR issues round out the findings.

---

## Findings

### [SERIOUS] Unbounded `data` payload enables memory exhaustion despite MAX_QUEUE_SIZE

- **File**: `apps/session-orchestrator/src/index.ts`
- **Line**: 153–155 (Zod schema) and 164–169 (enqueue call)
- **Issue**: `data` is validated as `z.record(z.string(), z.unknown()).optional()` with no size constraint on keys, values, or total payload. `MAX_QUEUE_SIZE=1000` limits event _count_, but each event can carry an arbitrarily large payload. A single caller passing `{ bigkey: "A".repeat(10_000_000) }` per event could push 10 GB into the in-process queue before the count cap triggers. There is no per-key length limit, no total-payload-bytes limit, and no nesting depth limit.
- **Fix**: Add a runtime byte-budget check before enqueuing (e.g., reject or truncate events where `JSON.stringify(data).length > 8192`). Additionally, restrict `z.record(z.string(), z.unknown())` to `z.record(z.string().max(64), z.string().max(512))` or a union of primitives, dropping `unknown` entirely. This removes the attack surface while still satisfying the documented use cases (`{ "task_id": "TASK_XXX" }`, `{ "task_id": "...", "batch": N }`).

### [SERIOUS] No caller binding on `worker_id` — any caller can emit events for any worker

- **File**: `apps/session-orchestrator/src/index.ts`
- **Line**: 156–178 (`emit_event` handler)
- **Issue**: The `worker_id` argument is accepted without verifying that it belongs to the calling session. Any MCP client (including a worker that has learned another worker's ID from `list_workers` output) can call `emit_event(worker_id="WID_other", label="IMPLEMENTED", ...)` and inject a fake phase-transition event into the supervisor's queue. Because the supervisor relies on these events for state machine decisions (spawning Review Workers, declaring stuck detection thresholds), a forged `IMPLEMENTED` event for an in-progress worker would cause the supervisor to prematurely spawn a Review Worker, corrupting task state. The check at line 159 only validates _existence_, not ownership.
- **Fix**: MCP tools do not have per-caller session identity available at the transport layer, so a hard cryptographic bind is not feasible without protocol changes. The pragmatic mitigation is a format allowlist: validate that `worker_id` matches the expected `WID_[a-z0-9]{8}` pattern (`/^WID_[a-z0-9]{8}$/`) before enqueuing. This ensures only IDs generated by `spawn_worker` are accepted and blocks arbitrary strings. Document that full authentication is deferred to a future protocol-level capability.

### [MINOR] Log injection via unsanitized `worker_id` and `event_label` written to stderr

- **File**: `apps/session-orchestrator/src/index.ts` line 161, `libs/worker-core/src/core/event-queue.ts` line 15–18
- **Issue**: Both files write `worker_id` and `event_label` directly into `process.stderr.write(...)` template literals without stripping newlines, carriage returns, or ANSI escape sequences. A `worker_id` containing `\n[emit_event] worker WID_admin found — elevated access granted` would inject a convincing false log line into stderr, which may be captured and parsed by monitoring tools.
  - `index.ts:161`: `` `[emit_event] worker ${worker_id} not found...` ``
  - `event-queue.ts:16–17`: `` `[event-queue] queue full...dropping event ${event.event_label} for ${event.worker_id}` ``
- **Fix**: Sanitize both fields before interpolation into log strings. Minimal fix: replace `\n`, `\r`, and ESC (`\x1b`) with a space or `<LF>` literal. Example: `worker_id.replace(/[\r\n\x1b]/g, '<LF>')`. Since `label` already has `.max(64)`, the same character sanitization should apply to it as well.

### [MINOR] `worker_id` has no maximum length bound

- **File**: `apps/session-orchestrator/src/index.ts`
- **Line**: 152 (Zod schema, `worker_id: z.string()`)
- **Issue**: `label` correctly enforces `.max(64)` but `worker_id` has no length constraint. An unbounded `worker_id` can be passed (e.g., a 1 MB string), which would be stored in the queue entry, written to stderr, and returned in the response — contributing to memory and log-size issues. The fix to finding #2 (format allowlist) implicitly resolves this, but the Zod schema should also enforce `.max(32)` independently as a defense-in-depth measure even if the regex check is added.
- **Fix**: Add `.max(32)` to the `worker_id` Zod field: `z.string().max(32).describe(...)`. The generated `WID_xxxxxxxx` format is 12 characters; 32 is a generous upper bound.

---

### [MINOR] `as` type assertion violates project convention

- **File**: `apps/session-orchestrator/src/index.ts`
- **Line**: 168
- **Issue**: `data: data as Record<string, unknown> | undefined` uses a type assertion (`as`), which is explicitly forbidden by the project's TypeScript conventions ("No `as` type assertions — if the type system fights you, the type is wrong"). Zod's inferred type for `z.record(z.string(), z.unknown()).optional()` is already `Record<string, unknown> | undefined`, so the assertion is unnecessary — the compiler accepts `data` directly without it.
- **Fix**: Remove the `as` cast: change `data: data as Record<string, unknown> | undefined` to `data: data`. No runtime change; this is purely a type-level cleanup.
