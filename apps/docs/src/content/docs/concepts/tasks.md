Every unit of work in Nitro-Fueled is a **task** — a folder containing a `task.md` definition file and the artifacts produced by each pipeline phase. Tasks are the primary communication mechanism between you, the Planner, and the Supervisor.

---

## What a Task Is

A task maps to a single unit of implementable work that fits within one worker session. The rule of thumb: if a task would require more than 7 files, more than 5 acceptance criteria groups, or spans multiple architectural layers at once, it is too large and should be split.

Each task lives in its own folder:

```
task-tracking/
  TASK_2026_001/
    task.md                 # Task definition (you write this)
    status                  # Current state: CREATED, IN_PROGRESS, etc.
    context.md              # Orchestrator scope document (Phase 0)
    task-description.md     # Project Manager requirements (Phase 1)
    implementation-plan.md  # Architect design (Phase 2)
    tasks.md                # Team-Leader batch breakdown (Phase 3)
    completion-report.md    # Build Worker final summary
    code-logic-review.md    # Logic review output
    code-style-review.md    # Style review output
    code-security-review.md # Security review output
```

## Task State Machine

Each task moves through a defined sequence of states. The current state is stored in a plain text file called `status` inside the task folder — one word, no metadata.

```
                   ┌─────────────────────────────────────────────────────┐
                   │                                                     │
                   ▼                                                     │
CREATED ──► IN_PROGRESS ──► IMPLEMENTED ──► IN_REVIEW ──► COMPLETE      │
   │              │                              │                       │
   │              ▼                              ▼                       │
   │            FAILED ◄──────────────────── FAILED                     │
   │                                                                     │
   ├──► BLOCKED ──► (back to CREATED when dependency resolves) ─────────┘
   │
   └──► CANCELLED
```

| State | Meaning |
|-------|---------|
| `CREATED` | Task defined, waiting to be picked up by the Supervisor |
| `IN_PROGRESS` | Build Worker is actively implementing |
| `IMPLEMENTED` | Build Worker finished, code committed, waiting for review |
| `IN_REVIEW` | Review Worker is running logic, style, and security checks |
| `COMPLETE` | All reviews passed, task fully done |
| `FAILED` | Worker failed or reviewer found blocking issues |
| `BLOCKED` | Waiting for a dependency task to reach `COMPLETE` |
| `CANCELLED` | Task abandoned, folder preserved for traceability |

## The task.md Format

```markdown
# Task: [Title]

## Metadata

| Field      | Value  |
|------------|--------|
| Type       | FEATURE |
| Priority   | P1-High |
| Complexity | Medium  |

## Description

[What needs to be built, fixed, or changed. At least 2 sentences.
The Project Manager reads this as the primary input for requirements.]

## Dependencies

- TASK_2026_003 — database schema must exist before this task runs
- None

## Acceptance Criteria

- [ ] [Verifiable outcome 1]
- [ ] [Verifiable outcome 2]

## References

- [Files, docs, or links the Architect and Developer should read]
```

## Task Registry

`task-tracking/registry.md` is an auto-generated index of all tasks. The Supervisor reads it to build the dependency graph and find actionable tasks. The registry is regenerated by `npx @itqanlab/nitro-fueled status` or `/project-status` — it should not be edited manually.

Example registry entry:

```markdown
| TASK_2026_001 | Add user greeting | FEATURE | P2-Medium | COMPLETE |
| TASK_2026_002 | Fix sidebar collapse | BUGFIX | P2-Medium | CREATED |
```

> **Note:** The `status` column in `registry.md` is informational. The `status` file inside each task folder is the authoritative source of truth for the Supervisor.

## plan.md — The Planner's Roadmap

`task-tracking/plan.md` is written and maintained exclusively by the `nitro-planner` agent. It contains:

- **Phases and milestones** — high-level roadmap structure
- **Task Map** — which tasks belong to which phase
- **Current Focus** — the section the Supervisor reads to decide what to run next

The Supervisor reads `plan.md` before each scheduling cycle. If two tasks have equal priority, `plan.md` ordering is used to break the tie.

## Task Types and Pipeline Routes

The `Type` field in `task.md` determines which agent pipeline the Build Worker runs:

| Type | Agent Flow |
|------|-----------|
| `FEATURE` | PM → Architect → Team-Leader → Developer → Review Lead + Test Lead |
| `BUGFIX` | [Research] → Team-Leader → Developer → Review Lead + Test Lead |
| `REFACTORING` | Architect → Team-Leader → Developer → Review Lead + Test Lead |
| `DOCUMENTATION` | PM → Developer → Style Reviewer |
| `RESEARCH` | Researcher → conditional implementation |
| `DEVOPS` | PM → Architect → DevOps Engineer → QA |
| `CREATIVE` | UI/UX Designer → Content Writer → Frontend Developer |

---

## Priority Levels

| Priority | Label | Meaning |
|----------|-------|---------|
| `P0-Critical` | Critical | Blocker — must run before anything else |
| `P1-High` | High | Core feature or urgent fix |
| `P2-Medium` | Medium | Standard work |
| `P3-Low` | Low | Nice-to-have, run when queue is clear |

The Supervisor orders the queue by priority (P0 first, P3 last) and uses `plan.md` ordering as a tiebreaker within the same priority level.
